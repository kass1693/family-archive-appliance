- 개요: 사진 업로드 기능에서 서버가 수행해야 하는 동작을 정의한다.

---

## 배경·목표 (Background & Goals)
- 배경: 클라이언트로부터 사진 파일을 수신하고, 저장·처리 후 영구 보존 가능한 상태로 관리해야 한다.
- 목표:
  - 클라이언트의 중복 확인 요청에 신뢰성 있게 응답할 수 있어야 한다.
  - 업로드된 파일을 지정된 경로에 원자적으로 저장하고 메타데이터를 기록해야 한다.
  - 원본 파일은 변경되지 않아야 하며, 서비스용 변환 파일은 별도로 생성해야 한다.

---

## 현재/목표 동작 (Current / Target Behavior)
- 현재 동작: 없음 (신규 구현)
- 목표 동작: 서버는 hash 기반 중복 확인 응답, 파일 수신 및 저장, EXIF 파싱, 변환 파일 생성의 전체 흐름을 수행한다.

---

## 가정·제약 (Assumptions & Constraints)
- 가정:
  - 파일은 multipart/form-data 형식으로 전송된다.
  - hash 계산은 클라이언트가 수행하며, 서버는 전달받은 hash를 저장된 데이터와 비교한다.
  - 원본 파일 저장 경로는 `originals/YYYY/MM/DD/{UUID}.{ext}` 형식을 따른다.
- 제약:
  - 원본 파일은 저장 후 수정되어서는 안 된다.
  - 중복 확인은 정보 제공 목적이며 서버는 클라이언트의 업로드 요청을 hash 중복만으로 거부하지 않는다.
  - 변환 파일 생성 실패는 업로드 전체를 실패로 처리하지 않는다.

---

## 범위·요구사항 (Scope & Requirements)
- 범위:
  - 포함: hash 기반 중복 확인 응답, 파일 수신 및 저장, EXIF 파싱 및 메타데이터 저장, 변환 파일 생성
  - 제외: 클라이언트 UI, 중복 여부에 따른 업로드 거부

### 기능 요구사항 (Functional Requirements)

#### REQ-F-001: 중복 확인 응답
- 설명: 클라이언트는 업로드 전 파일별 hash를 계산해 서버에 전달하고, 서버는 이미 저장된 파일 중 동일한 hash가 있는 것을 반환한다. 이를 통해 클라이언트는 중복 파일 여부를 사전에 확인할 수 있다.
- 근거/목적: 클라이언트가 업로드 전에 중복 여부를 인지할 수 있도록 하여 불필요한 전송을 줄인다.
- 수용 기준:
  - 서버는 전달된 hash 목록 중 DB에 동일한 hash로 저장된 항목을 반환한다.
  - 중복이 없을 경우 빈 목록을 반환한다.
- 예외/경계 조건:
  - 중복이라도 업로드가 가능해야 한다. 서버는 중복 hash를 이유로 이후 업로드 요청을 거부하지 않는다.

#### REQ-F-002: 파일 수신 및 저장
- 설명: 서버는 클라이언트로부터 파일을 수신하고 `originals/YYYY/MM/DD/{UUID}.{ext}` 경로에 저장한다. 먼저 `import/pending/` 임시 경로에 저장 후 최종 경로로 이동한다.
- 근거/목적: 임시 저장 후 이동하는 방식으로 저장 도중 장애가 발생해도 부분 파일이 최종 경로에 남지 않도록 한다.
- 수용 기준:
  - 파일은 최종적으로 `originals/YYYY/MM/DD/{UUID}.{ext}` 에 위치한다.
  - 파일명은 UUID로 생성되어 충돌 가능성이 없어야 한다.
  - 임시 경로에서 최종 경로로 이동이 완료된 이후에만 성공으로 간주한다.
- 예외/경계 조건:
  - 최종 경로 이동 전 장애가 발생하면 임시 파일은 정리되어야 한다.

#### REQ-F-003: EXIF 파싱 및 메타데이터 저장
- 설명: 서버는 저장된 원본 파일에서 EXIF 메타데이터를 파싱하고 DB에 기록한다. 저장 항목에는 파일명, UUID, hash, 업로드 시각, 촬영 시각, EXIF 원본 데이터가 포함된다.
- 근거/목적: 날짜 기반 탐색, 중복 확인, 파일 무결성 검증에 메타데이터가 필요하다.
- 수용 기준:
  - EXIF에서 파싱 가능한 모든 필드가 DB에 그대로 저장된다.
  - EXIF가 없는 파일의 경우 파싱 오류 없이 빈 EXIF 필드로 저장된다.
  - DB 레코드는 `originals/` 경로 이동 완료 후 생성된다.
- 예외/경계 조건:
  - EXIF 파싱 실패가 업로드 전체를 실패로 만들어서는 안 된다. 파싱 실패 시 EXIF 필드는 비워 저장한다.

#### REQ-F-004: 변환 파일 생성
- 설명: 서버는 원본 파일로부터 thumbnail과 medium 크기의 변환 파일을 생성하여 별도 경로에 저장한다.
- 근거/목적: 브라우저에서 빠르게 렌더링할 수 있도록 원본과 분리된 서비스용 파일이 필요하다.
- 수용 기준:
  - thumbnail과 medium 변환 파일이 생성된다.
  - 변환 파일은 원본과 별도 경로에 저장된다.
  - 변환 파일의 파일명은 원본 UUID를 기반으로 한다.
- 예외/경계 조건:
  - 변환 파일 생성 실패는 업로드 전체를 실패로 처리하지 않는다. 단, 재생성이 필요함을 추적할 수 있어야 한다.

### 비기능 요구사항 (Non-Functional Requirements)

#### REQ-NF-001: 업로드 원자성
- 품질 속성: 신뢰성 (Reliability)
- 설명: 파일 저장, EXIF 파싱, DB 기록의 전체 흐름이 완료된 경우에만 업로드 성공으로 간주한다. 일부 단계만 완료된 상태로 시스템이 남아서는 안 된다.
- 측정/검증 기준: 중간 단계에서 강제 종료 시 임시 파일은 제거되고 DB 레코드는 존재하지 않는다.

#### REQ-NF-002: 부분 실패 시 상태 일관성
- 품질 속성: 일관성 (Consistency)
- 설명: 파일 저장 성공 후 DB 기록이 실패한 경우, 저장된 파일을 제거하여 파일시스템과 DB가 불일치 상태로 남지 않아야 한다.
- 측정/검증 기준: DB 기록 실패 시 `originals/` 에 저장된 파일이 제거된다.

#### REQ-NF-003: 원본 파일 불변성
- 품질 속성: 무결성 (Integrity)
- 설명: `originals/` 경로에 저장된 파일은 이후 어떤 처리에서도 수정되거나 덮어쓰여서는 안 된다.
- 측정/검증 기준: 저장 시 기록된 SHA-256 hash와 이후 파일의 hash가 일치한다.
