- 개요: 사진 업로드 기능에서 클라이언트가 수행해야 하는 동작을 정의한다.

---

## 배경·목표 (Background & Goals)
- 배경: 홈서버에 사진을 저장하기 위해 클라이언트에서 파일을 선택하고 서버로 전송하는 흐름이 필요하다.
- 목표:
  - 클라이언트가 중복 파일을 서버에 전송하기 전에 사전 확인할 수 있어야 한다.
  - 사용자가 중복 여부를 인지하고 업로드 여부를 결정할 수 있어야 한다.
  - 파일 선택부터 업로드 요청까지의 흐름이 명확하게 정의되어야 한다.

---

## 현재/목표 동작 (Current / Target Behavior)
- 현재 동작: 없음 (신규 구현)
- 목표 동작: 클라이언트는 파일을 선택하고, 각 파일의 hash를 계산하여 서버에 중복 확인을 요청한다. 결과를 사용자에게 표시하고, 사용자가 최종 업로드 여부를 결정한다.

---

## 가정·제약 (Assumptions & Constraints)
- 가정:
  - 클라이언트는 브라우저 환경에서 동작한다.
  - SHA-256 hash 계산은 브라우저에서 Web Crypto API로 수행 가능하다.
  - 서버는 중복 확인 API와 파일 업로드 API를 별도로 제공한다.
- 제약:
  - 클라이언트는 서버 내부 저장 구조(경로, DB 구조)를 알지 않는다.
  - 중복 확인 결과는 최종 판단 정보로만 사용하며, 서버가 최종 거부 여부를 결정하지 않는다.

---

## 범위·요구사항 (Scope & Requirements)
- 범위:
  - 포함: 파일 선택, hash 계산, 중복 확인 요청, 결과 표시, 업로드 요청
  - 제외: 서버 측 파일 저장, EXIF 파싱, 변환 파일 생성

### 기능 요구사항 (Functional Requirements)

#### REQ-F-001: 파일별 hash 계산
- 설명: 클라이언트는 사용자가 선택한 각 파일에 대해 SHA-256 hash를 계산한다. 이 hash는 중복 여부를 사전에 확인하기 위한 식별자로 사용된다.
- 근거/목적: 파일 내용이 동일한지를 서버에 전달하기 위해 내용 기반 식별자가 필요하다.
- 수용 기준:
  - 선택된 모든 파일에 대해 SHA-256 hash가 계산된다.
  - hash 계산은 파일 내용 전체를 대상으로 한다.
- 예외/경계 조건:
  - 파일 크기가 매우 큰 경우 hash 계산 중 UI가 응답하지 않아서는 안 된다.

#### REQ-F-002: 중복 확인 요청
- 설명: 클라이언트는 계산된 hash 목록을 서버에 전달하고, 서버로부터 이미 저장된 파일 중 동일한 hash를 가진 항목 목록을 받는다.
- 근거/목적: 업로드 전에 중복 파일을 식별하여 사용자가 불필요한 전송을 피할 수 있도록 한다.
- 수용 기준:
  - 중복 확인 요청은 파일 전송 이전에 수행된다.
  - 서버 응답에서 중복으로 확인된 파일과 그렇지 않은 파일을 구분할 수 있다.
- 예외/경계 조건:
  - 서버 응답이 실패할 경우 사용자에게 재시도 할 수 있는 선택지를 제공한다.

#### REQ-F-003: 중복 결과 표시 및 업로드 여부 결정
- 설명: 클라이언트는 중복 확인 결과를 사용자에게 표시하고, 사용자가 각 파일의 업로드 여부를 결정할 수 있도록 한다.
- 근거/목적: 중복 파일을 자동으로 제외하거나 강제하면 사용자의 의도와 다른 결과가 발생할 수 있다. 최종 결정권은 사용자에게 있어야 한다.
- 수용 기준:
  - 중복 파일과 새 파일이 구분되어 표시된다.
  - 사용자는 중복 파일에 대해 업로드 여부를 선택할 수 있다.
  - 사용자가 선택을 확인한 이후 업로드가 시작된다.
- 예외/경계 조건:
  - 모든 파일이 중복인 경우에도 사용자가 업로드를 선택할 수 있다.

#### REQ-F-004: 파일 업로드 요청
- 설명: 클라이언트는 사용자가 업로드하기로 결정한 파일을 서버에 전송한다.
- 근거/목적: 실제 파일 저장은 서버에서 수행되므로 클라이언트는 결정된 파일만 서버로 전달한다.
- 수용 기준:
  - 사용자가 선택한 파일만 전송된다.
  - 전송 중 진행 상태가 사용자에게 표시된다.
  - 서버 응답에 따라 성공 또는 실패 상태가 사용자에게 표시된다.
- 예외/경계 조건:
  - 네트워크 오류 발생 시 사용자에게 오류를 알리고 재시도 선택지를 제공한다.
  - 개별 파일 업로드 실패가 다른 파일 업로드에 영향을 주어서는 안 된다.

### 비기능 요구사항 (Non-Functional Requirements)

#### REQ-NF-001: hash 계산 응답성
- 품질 속성: 응답성 (Responsiveness)
- 설명: 파일 선택 후 hash 계산 중에도 UI는 사용자 인터랙션에 응답할 수 있어야 한다.
- 측정/검증 기준: 100MB 파일 hash 계산 중 페이지가 멈추지 않고 진행 상태를 표시한다.

#### REQ-NF-002: 상태 표시 명확성
- 품질 속성: 사용성 (Usability)
- 설명: hash 계산 중, 중복 확인 요청 중, 업로드 진행 중 각 단계의 상태가 사용자에게 명확하게 표시되어야 한다.
- 측정/검증 기준: 각 단계별 진행 상태 표시가 있고, 완료 또는 오류 여부를 사용자가 인식할 수 있다.
