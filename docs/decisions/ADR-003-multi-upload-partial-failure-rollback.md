## 맥락 (Context)

### 문제 상황
클라이언트는 한 번의 요청으로 여러 파일을 업로드할 수 있다. 이미 REQ-NF-004에서 "하나라도 저장 실패 시 전체 요청을 실패 응답으로 처리한다"고 결정되었다. 단일 파일 기준 원자성(파일시스템 저장 + DB 기록)은 Usecase가 보장한다.

결정이 남은 것은 다중 파일 처리 중 일부 파일이 이미 저장 완료(파일시스템 + DB)된 상태에서 다른 파일이 실패했을 때, 이미 저장된 파일을 롤백할지 여부다.

### 목표
- 실패 응답의 의미론이 응답 레벨과 저장 레벨에서 일치해야 한다.
- 클라이언트가 재시도 시 예측 가능한 상태를 보장해야 한다.

### 제약 조건
- **기술적 제약:** 파일시스템과 MongoDB는 하나의 트랜잭션으로 묶을 수 없다. Usecase의 수동 복구에 의존한다.
- **조직적 제약:** 단일 운영자. 구현 복잡도가 높아지더라도 구조적 정확성을 우선한다.

### 관련 요구사항
- UPLOAD-SERVER-SRS-REQ-NF-004: 다중 파일 업로드 실패 정책

---

## 결정 (Decision)

### 최종 선택된 방안
**롤백 (전체 원자성)**

다중 파일 업로드 중 하나라도 실패하면, 해당 요청에서 이미 저장 완료된 파일을 파일시스템과 DB 모두에서 삭제한다.

### 정책 요소 (Policy Breakdown)
- **적용 범위:** 단일 요청 내 다중 파일 업로드
- **핵심 규칙:**
  - 롤백 로직은 Usecase 레이어에 위치한다. Presentation(Controller)은 각 파일의 Usecase 호출 결과를 수집할 뿐이며, 롤백 판단과 실행은 하지 않는다.
  - 파일 처리는 순차적으로 수행한다. 실패 발생 시 이후 파일 처리를 중단하고 이미 성공한 파일을 역순으로 롤백한다.
  - 롤백 실패(삭제 불가)는 에러 로그에 경로를 포함하여 기록한다. 수동 정리가 가능하도록 충분한 정보를 남긴다.
- **예외 규칙:** 없음

---

## 근거 (Rationale)

### 선택한 방안의 장점
- `docs/architecture/software/overview.md` 3.2절이 rollback을 Usecase의 명시적 책임으로 정의하고 있다. 롤백을 적용하지 않으면 아키텍처 문서와 구현이 즉시 충돌한다.
- REQ-NF-004가 "전체 요청 실패"를 정의하는데, 롤백 없이는 실패 응답을 반환하면서 일부 파일이 DB와 파일시스템에 잔류하는 모순이 생긴다.
- 클라이언트가 재시도 시 "어느 파일이 이미 저장됐는가"를 별도로 확인할 필요가 없어 재시도 흐름이 단순하다.

### 고려했으나 채택하지 않은 대안

- **롤백 없음**
  - 장점: 구현 단순, 성공한 파일은 보존됨.
  - 단점: 전체 실패 응답을 반환하면서 일부 파일이 저장된 상태가 된다. 응답 레벨의 실패 의미론과 저장 레벨의 상태가 불일치한다. 클라이언트 재시도 시 중복 업로드 가능성이 생기며, 중복 확인 UX가 이를 보완하더라도 근본적인 불일치를 덮는 구조다.

---

## 결과 (Consequences)

### 긍정적 효과
- "전체 요청 실패"의 의미론이 응답과 저장 레벨 모두에서 일관된다.
- 클라이언트 재시도 시 상태가 항상 예측 가능하다.

### 부정적 효과 및 위험
- 이미 성공한 파일 목록을 추적하다가 실패 시 역순으로 롤백하는 구현이 Usecase에 추가된다.
- 롤백 도중 파일 삭제 실패 시 고아 파일이 발생할 수 있다. 에러 로깅으로 수동 정리가 가능하도록 보완한다.

### 재검토 조건
- 업로드 파일 수가 매우 많아질 경우 롤백 비용이 실질적인 문제가 될 때.
