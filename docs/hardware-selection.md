# 하드웨어 선택 검토

이 문서는 `family-archive-appliance` 프로젝트의 하드웨어 선택 기준과 현재까지의 검토 내용을 정리합니다.

아직 구매가 확정된 부품은 없습니다. 실제 구매 후에는 이 문서를 업데이트합니다.

---

## 1. 배경

이 프로젝트의 주 목적은 가족 사진 아카이브 서버를 직접 만드는 것입니다. 부 목적은 Linux, Docker Compose, Web API, 스토리지 정책, 백업 정책을 포함한 개인용 홈서버 플랫폼을 확보하는 것입니다.

초기에는 완제품 NAS 구매도 고려했습니다. 하지만 Docker 기반 커스텀 서비스 운영 자유도, 성능 확장 가능성, 직접 구성하는 학습 가치를 고려해 자작 홈서버 방향으로 검토 중입니다.

---

## 2. 완제품 NAS와 자작 NAS 비교

### 완제품 NAS

장점:

- 가격과 완성도 면에서 유리한 편
- 관리 UI, 사진 앱, 백업 기능이 이미 제공됨
- 전력, 소음, 디스크 관리가 비교적 안정적
- 가족이 바로 사용하기 쉬움

단점:

- CPU/RAM 성능이 낮음
- Docker 기반 커스텀 서비스 운영에 제약이 있음
- ARM SoC 기반 제품은 Linux/Docker 개발 자유도가 낮음
- 확장성이 제한됨

### 자작 NAS

장점:

- N100/N150 기반 x86 보드를 사용할 수 있음
- Docker Compose 기반 서비스 구성이 자유로움
- Web API, Web UI, worker, 모니터링 등을 직접 설계 가능
- RAM, NVMe, SATA, LAN 확장성이 좋음
- 개인 사이드 프로젝트 플랫폼으로 활용 가능

단점:

- 완제품 NAS보다 총액이 높아질 수 있음
- 하드웨어 호환성 확인이 필요함
- OS 설치, 디스크 마운트, 백업, 운영 정책을 직접 책임져야 함
- HDD를 포함하면 예산 부담이 커짐

### 현재 판단

패밀리 앨범만 빠르게 구축하려면 완제품 NAS가 합리적인 선택이다. 하지만 Docker 기반 홈서버 플랫폼, 사이드 프로젝트 환경, 직접 구성하는 재미까지 고려하면 자작 방향도 충분히 의미가 있다. 이 프로젝트는 후자에 가깝다.

---

## 3. 예산 구조

전체 예산은 대략 80만 원 전후를 기준으로 잡고 있습니다.

HDD를 포함하면 자작 NAS는 완제품 NAS보다 비싸질 가능성이 높습니다. 따라서 예산을 본체와 스토리지로 분리해서 봅니다.

**본체 예산:**

- 보드
- 케이스
- PSU
- RAM
- OS용 NVMe SSD

**스토리지 예산 (단계적 추가):**

- 데이터 HDD (초기)
- 백업 HDD (후속 단계)
- 외장 백업 HDD (후속 단계)

초기 목표는 본체를 먼저 구성하고, 단일 HDD로 MVP를 시작하는 것입니다. 스토리지는 예산 상황에 따라 단계적으로 확장합니다.

---

## 4. CPU/SoC 선택 기준

### N100/N150 계열

N100과 N150은 일반 소켓 CPU가 아니라 보드에 실장된 저전력 SoC 계열입니다.

- N100은 패밀리 앨범 서버와 Docker 기반 홈서버 용도로 충분함
- N150은 N100의 소폭 개선판 정도로 봄
- 같은 보드 구성과 비슷한 가격이면 N150이 낫지만, 검증된 N100 보드가 더 싸다면 N100도 충분함

### ARM SoC 배제 이유

RTD1619B/RTD1619D 같은 완제품 NAS용 ARM SoC도 검토했습니다.

완제품 NAS 안에서는 적절하지만, 이 프로젝트에는 적합하지 않다고 판단했습니다.

이유:

- BSP, 부트로더, 커널, 드라이버 의존성이 큼
- Docker 이미지와 Linux 운영 자유도가 x86보다 낮음
- 이 프로젝트의 목적은 보드 포팅이 아니라 appliance 구성임

---

## 5. 보드 선택 기준

현재 가장 유력한 방향은 N100 기반 Mini-ITX NAS 보드입니다. CWWK 또는 유사 계열 보드를 검토 중이며, 가격은 방열판 포함 약 35만 원 수준입니다.

### 필수 조건

| 항목 | 기준 |
|---|---|
| CPU | x86 기반 N100 또는 N150 |
| 폼팩터 | Mini-ITX 또는 NAS 케이스 장착 가능 |
| SATA | 4개 이상 (가능하면 6개) |
| M.2 NVMe | 1개 이상 |
| RAM | DDR5 또는 DDR4 SO-DIMM, 8GB 이상 장착 가능 |
| NIC | Intel i226-V 또는 Linux에서 안정 지원되는 NIC |
| 전원 | ATX/SFX PSU 연결 가능 (24핀 + 4핀 ATX 12V) |
| BIOS | 설정 접근 가능, 전원 복구 옵션 존재 |

### 선호 조건

- SATA 6개
- M.2 NVMe 2개
- 2.5GbE 2포트 이상
- 사용자 후기 또는 Linux 설치 사례 존재
- BIOS 다운로드 또는 업데이트 경로 존재
- 케이스 호환성 사례 존재
- JONSBO radiator 또는 적절한 방열판 옵션 존재

### 주의 조건

다음에 해당하면 신중하게 검토합니다.

- DC Power 방식인데 HDD 전원 공급 구조가 불명확한 경우
- M.2 슬롯 사용 시 SATA 포트 일부가 비활성화되는 경우
- 보드 리비전이 불명확하거나 변경 이력이 없는 경우
- BIOS 다운로드 경로가 없는 경우
- 후기가 지나치게 적거나 전원 안정성 관련 부정 후기가 있는 경우
- 판매자와 실제 제조사가 분리되어 있어 AS 경로가 불명확한 경우

---

## 6. 전원 구조 판단

NAS 구성에서는 CPU 전력보다 HDD 스핀업 전류와 SATA 전원 안정성이 더 중요합니다.

### DC Power 방식

소형 보드에 주로 사용되며 크기와 비용 면에서 유리합니다. 하지만 3.5인치 HDD를 여러 개 구동할 때 전원 공급 경로가 명확하지 않은 경우가 있습니다. 사용 전 반드시 HDD 전원 구조를 확인해야 합니다.

### ATX/SFX PSU 방식

이 프로젝트에서 선호하는 방향입니다.

전원 구조가 명확합니다.

- 24핀 ATX: 메인보드 전원
- 4핀 ATX 12V: CPU/보드 보조 전원
- SATA Power 커넥터: HDD 전원 (PSU에서 직접 공급)

3.5인치 HDD를 여러 개 사용할 가능성이 있다면 PSU 기반 구성이 더 안정적입니다.

---

## 7. 케이스 선택 기준

케이스는 아직 확정하지 않았습니다. 현재 검토 중인 후보는 다음과 같습니다.

- JONSBO N2
- JONSBO N3
- Fractal Node 304
- 기타 Mini-ITX NAS 케이스

### 판단 기준

| 항목 | 내용 |
|---|---|
| 보드 장착 | Mini-ITX 보드 장착 가능 |
| HDD 베이 | 3.5인치 2개 이상, 가능하면 4개 이상 확장 가능 |
| PSU | SFX 또는 ATX PSU 지원 여부 확인 |
| 냉각 | HDD 쪽 팬 장착 가능 여부, 냉각 구조 |
| 케이블 | SATA 케이블 라우팅, 보드 SATA 포트 위치와 간섭 여부 |
| 백플레인 | 백플레인 사용 여부 및 호환성 |
| 가격 | 예산 내 수용 가능 여부 |

NAS다운 형태의 JONSBO N2/N3도 매력적이지만, 예산을 고려하면 일반 ATX PSU를 사용할 수 있는 케이스도 검토 대상입니다.

---

## 8. RAM 선택 기준

초기에는 8GB와 16GB를 모두 후보로 둡니다.

**8GB:**

- 패밀리 앨범 MVP에는 충분할 가능성이 높음
- 비용 절감에 유리

**16GB:**

- Docker 서비스 확장에 유리
- 추가 서비스(모니터링, 이미지 처리 등) 운영 가능성에 대비 가능
- SO-DIMM 슬롯이 1개인 경우, 처음부터 16GB를 선택하는 편이 나중에 더 합리적임

기본값은 8GB로 두되, 가격 차이가 크지 않으면 16GB를 선택하는 것을 고려합니다.

---

## 9. 저장장치 선택 기준

### OS용 NVMe SSD

- 128GB 또는 256GB
- OS 설치, Docker 이미지, 로그 저장 용도
- 가격 차이가 크지 않으면 256GB를 선택하는 편이 낫다

### 데이터 HDD

초기에는 HDD 가격 부담 때문에 단일 HDD로 시작할 수 있습니다.

**단일 HDD에 관한 원칙:**

- 하나의 HDD를 파티션으로 나누는 것은 정리에는 도움이 되지만 백업이 아니다.
- 단일 HDD는 MVP 시작점으로만 허용한다.
- 백업 HDD 또는 외장 HDD가 추가되기 전까지 이 서버를 유일한 원본 저장소로 보지 않는다.
- 사진 원본은 휴대폰, 기존 PC, 클라우드 등 다른 위치에도 유지한다.
- 나중에 백업 HDD를 추가해도 경로 구조가 크게 바뀌지 않도록 설계한다.

**초기 구성:**

| 용도 | 장치 | 비고 |
|---|---|---|
| OS | M.2 NVMe SSD 128GB 또는 256GB | 부팅 드라이브 |
| Data | HDD 1개 | 사진 원본 저장 |
| Backup | 미정 | 추후 HDD 또는 외장 HDD 추가 |

## 10. 하드웨어 디버깅 범위

이 프로젝트에서 하드웨어 디버깅은 다음 수준으로 정의합니다.

**포함:**

- BIOS 설정 확인 및 업데이트
- SATA 포트 인식 확인
- NVMe 부팅 확인
- NIC 인식 확인
- 팬 속도 및 온도 센서 확인
- 전원 복구 옵션 확인
- Linux kernel 호환성 확인
- ASPM/C-State 등 전력 관리 설정 확인

**제외:**

- BIOS 직접 빌드 또는 coreboot 포팅
- 보드 회로 수정, 납땜, 하드웨어 리워크
- 제조사 BSP 수준의 커널 포팅

BIOS 빌드나 포팅은 가능하더라도 이 프로젝트의 초기 범위 밖으로 둡니다. 목적은 홈서버 appliance 구성이지 보드 포팅이 아닙니다.
